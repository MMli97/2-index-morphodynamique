## A1 — Profondeur hiérarchique

| Sous-critère                                  | Score | Justification                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --------------------------------------------- | ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| H1 : ≥ 2 niveaux causaux distincts            | 1     | Niveau 1 : matériel (registres, interruptions, DMA). Niveau 2 : sous-systèmes noyau (scheduler, VFS, memory manager). Deux niveaux causaux clairement distincts : le matériel déclenche des interruptions que le noyau traite, et le noyau commande le matériel via les drivers. [1, ch. 1-7]                                                                                                                                  |
| H2 : ≥ 3 niveaux causaux distincts            | 1     | Niveau 3 : interface syscall / espace utilisateur. Les appels système forment une frontière causale nette : les processus user-space invoquent des services noyau qui à leur tour interagissent avec les pilotes matériels. Trois niveaux : hardware → kernel subsystems → user-space interface. [1, ch. 5 ; 2, « syscall »]                                                                                                   |
| H3 : ≥ 4 niveaux causaux distincts            | 1     | Niveau 4 : processus de gouvernance et de maintenance (mailing lists, arbre de mainteneurs, cycles de release). Les décisions humaines de merge/revert modifient causalement l'architecture du code, qui à son tour modifie le comportement des sous-systèmes. On obtient : hardware → kernel subsystems → syscall/user interface → governance/maintenance process. [3 ; 4]                                                    |
| H4 : Niveaux fonctionnellement différenciés   | 1     | Chaque niveau remplit une fonction irréductible à celle des autres : le matériel exécute physiquement, les sous-systèmes noyau arbitrent les ressources (scheduling, mémoire, I/O), l'interface syscall fournit l'abstraction POSIX, la gouvernance assure la cohérence évolutive du projet. Pas de redondance fonctionnelle entre ces niveaux. [1, ch. 1 ; 3]                                                                 |
| H5 : Causalité bidirectionnelle entre niveaux | 1     | Exemples multiples : le scheduler adapte ses décisions en fonction de signaux matériels (fréquence CPU, NUMA topology) — causalité ascendante ; les politiques de merge des mainteneurs restructurent les sous-systèmes — causalité descendante ; les bugs observés en user-space déclenchent des patches dans les couches noyau et parfois des mises à jour firmware — boucle traversant tous les niveaux. [1, ch. 4 ; 3 ; 4] |

**Score A1 = 1.00 / 1.00**

**Hésitations / ambiguïtés :** La question se posait de savoir si le processus de gouvernance humaine compte comme un « niveau causal » du système ou comme un facteur externe. Étant donné l'énoncé (« code + architecture + processus de maintenance »), ce niveau est explicitement dans le périmètre. Par ailleurs, on pourrait argumenter un cinquième niveau (écosystème des distributions), mais cela sort du périmètre défini.

## A2 — Capacité de propagation (formulation V2.1.1 : invariance d'échelle)

| Sous-critère                                                  | Score | Justification                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------------------------------------------------- | ----- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P1 : Perturbation locale affecte ≥ 1 autre module fonctionnel | 1     | Un bug dans un pilote réseau peut provoquer un memory leak qui affecte le memory manager, puis le scheduler (OOM killer). Un changement dans l'allocateur slab se propage à tous les sous-systèmes consommateurs. La modularité est réelle mais les dépendances entre sous-systèmes sont denses. [1, ch. 12-15]                                                                                                                                                            |
| P2 : Propagation traverse ≥ 1 niveau hiérarchique             | 1     | Un défaut matériel (niveau hardware) déclenche une interruption traitée par le noyau, pouvant provoquer un kernel panic visible en user-space. Inversement, un syscall mal géré peut corrompre des structures internes du noyau. La propagation inter-niveaux est routinière. [1, ch. 7 ; 2, « oops »]                                                                                                                                                                     |
| P3 : Propagation modifie l'état global observable             | 1     | Un kernel panic ou un deadlock dans un sous-système critique (VFS, scheduler) rend l'ensemble du système inutilisable. Un bug de sécurité dans un seul syscall (ex. : Dirty COW) compromet l'intégrité de la machine entière. L'état global observable est modifié de manière flagrante. [4, articles sur vulnérabilités noyau]                                                                                                                                            |
| P4 : Isolement difficile sans modification structurelle       | 0.5   | Le noyau possède des mécanismes d'isolation (modules chargeables, namespaces, cgroups) qui permettent un certain confinement. Mais dans le code noyau lui-même (ring 0), l'isolation stricte est impossible sans modification architecturale majeure : tout module noyau a accès à l'espace d'adressage complet. Score 0.5 car l'isolation est partiellement possible (modules, namespaces) mais structurellement limitée en kernel-space. [1, ch. 17 ; 2, « namespaces »] |
| P5 : Couplage fonctionnel non trivial                         | 1     | Les interactions entre sous-systèmes passent par des mécanismes complexes : le VFS dépend du memory manager (page cache), qui dépend du scheduler (writeback threads), qui dépend des pilotes bloc. Ce couplage n'est pas une simple chaîne linéaire : il forme un réseau de dépendances avec des boucles. Les locking primitives (spinlocks, mutexes, RCU) témoignent de la complexité du couplage. [1, ch. 8-10]                                                         |

**Score A2 = 0.90 / 1.00**

**Hésitations / ambiguïtés :** P4 est le point de tension principal. Le noyau Linux est monolithique (tout tourne en ring 0), ce qui rend l'isolement structurellement difficile. Cependant, les mécanismes de modularité (modules chargeables, Kconfig) et d'isolation user-space (namespaces, cgroups, seccomp) offrent un confinement partiel. On ne peut pas dire que l'isolement est « impossible » (0 serait excessif), ni qu'il est pleinement « difficile sans modification structurelle » (car certaines modifications mineures comme le chargement/déchargement de modules suffisent). D'où 0.5.

## A3 — Intégration

| Sous-critère                             | Score | Justification                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ---------------------------------------- | ----- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| I1 : Mécanisme explicite de coordination | 1     | Multiples mécanismes formels : le scheduler coordonne l'accès CPU entre tous les processus ; les primitives de synchronisation (spinlocks, mutexes, semaphores, RCU) coordonnent l'accès concurrent aux structures de données ; les waitqueues et les signaux coordonnent les processus. Au niveau gouvernance : le processus de merge via l'arbre des mainteneurs est un mécanisme de coordination explicite et documenté. [1, ch. 4, 8-10 ; 3]                                                                                                                                                |
| I2 : Réduction de variance observable    | 1     | Le scheduler assure que le temps CPU est distribué de façon prédictible (CFS : « Completely Fair Scheduler »). Le memory manager réduit la variance d'accès mémoire via le page cache et le swap. Le processus de release suit un cycle strict (merge window de 2 semaines, puis RC hebdomadaires) qui réduit drastiquement la variance dans le rythme de développement. [1, ch. 4 ; 3]                                                                                                                                                                                                         |
| I3 : Synchronisation multi-niveaux       | 1     | Exemples : les interruptions matérielles sont synchronisées avec le code noyau via les mécanismes top-half / bottom-half (softirqs, tasklets, workqueues). Le scheduler synchronise les couches matérielles (CPU frequency scaling, CPU affinity) avec les demandes user-space. Le processus de release synchronise le travail de milliers de développeurs avec les contraintes de stabilité du code. [1, ch. 7-8 ; 3]                                                                                                                                                                          |
| I4 : Boucles de rétroaction globales     | 0.5   | Au niveau technique : le OOM killer est une boucle de rétroaction globale (pression mémoire → kill de processus → libération de ressources). Le système de congestion control réseau (TCP) ajuste le débit en réponse à la charge globale. Cependant, ces boucles sont spécialisées par sous-système ; il n'existe pas de boucle de rétroaction unique englobant le système entier. Au niveau gouvernance : les rapports de régression jouent ce rôle (bug → revert → stabilité), mais le couplage est lâche. Score 0.5 car les boucles existent mais restent sectorielles. [1, ch. 12, 15 ; 4] |
| I5 : Maintien d'un état global cohérent  | 1     | Le noyau maintient en permanence un état global cohérent : la table des processus, l'arbre VFS, les tables de routage réseau, les structures mémoire — tout cela forme un état global qui doit rester consistant sous accès concurrent massif. Le kernel « tourne » ou il panique : il n'y a pas d'état intermédiaire dégradé acceptable pour les structures critiques. Les mécanismes RCU et les memory barriers existent précisément pour garantir cette cohérence. [1, ch. 8-10 ; 2, « RCU »]                                                                                                |

**Score A3 = 0.90 / 1.00**

**Hésitations / ambiguïtés :** I4 est le point délicat. Le noyau Linux n'a pas de « boucle de rétroaction globale » unique et centralisée qui surveille l'état de tous les sous-systèmes simultanément — ce n'est pas un thermostat. Les rétroactions sont distribuées et spécialisées (OOM killer pour la mémoire, congestion control pour le réseau, deadline scheduler pour les I/O). On pourrait argumenter 1 en considérant que la combinaison de ces boucles produit un effet de rétroaction globale émergente, mais chacune prise isolément reste sectorielle.

## A4 — Normativité

| Sous-critère                                 | Score | Justification                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| -------------------------------------------- | ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| N1 : Attracteur dynamique existant           | 1     | Le noyau tend vers un état stable de fonctionnement : processus ordonnancés, mémoire allouée/libérée, I/O traitées. Après une perturbation (pic de charge, défaillance matérielle transitoire), le système converge vers son régime nominal. Le cycle de release (merge window → RCs → release) est un attracteur temporel très régulier côté gouvernance. [1, ch. 1 ; 3]                                                                                                                                                                                                                            |
| N2 : Correction active d'écart               | 1     | Le scheduler corrige activement les déséquilibres de charge CPU (load balancing). Le memory manager déclenche le page reclaim quand la mémoire libre descend sous un seuil. Le watchdog détecte les soft-lockups. Côté gouvernance : la règle « no regressions » de Torvalds impose le revert de tout patch qui casse un comportement existant — correction active explicite. [1, ch. 4, 12 ; 3 ; 4]                                                                                                                                                                                                 |
| N3 : Hiérarchie de priorités régulatoires    | 1     | Le noyau implémente des priorités explicites : les interruptions ont priorité sur les softirqs, qui ont priorité sur les processus normaux. Le scheduler distingue les classes de scheduling (SCHED_FIFO > SCHED_RR > SCHED_NORMAL). L'OOM killer priorise les processus à tuer. Côté gouvernance : la stabilité prime sur les nouvelles fonctionnalités, la sécurité prime sur la performance — hiérarchie documentée. [1, ch. 4, 7 ; 3]                                                                                                                                                            |
| N4 : Mécanisme interne de stabilisation      | 1     | Mécanismes multiples : le rate limiting (printk_ratelimit, CFS bandwidth throttling), les back-off exponentiels réseau, les mécanismes de flow control dans le block layer. Le cycle RC (Release Candidate) est un mécanisme de stabilisation délibéré : chaque semaine de RC réduit le taux de changement autorisé jusqu'à convergence. [1, ch. 4, 15 ; 3]                                                                                                                                                                                                                                          |
| N5 : Résistance aux perturbations prolongées | 0.5   | Le noyau résiste bien aux perturbations transitoires (pics de charge, perte temporaire de réseau). Cependant, sous perturbation prolongée (fuite mémoire continue, saturation disque persistante, attaque DoS soutenue), la dégradation est réelle même si progressive. Le noyau n'a pas de mécanisme d'auto-réparation fondamentale : un kernel compromise reste compromis jusqu'au reboot. Côté gouvernance, la résilience est meilleure : le projet a survécu à des conflits majeurs, des forks, et des pressions commerciales sur 30+ ans. Score 0.5 pour refléter cette dualité. [1, ch. 1 ; 4] |

**Score A4 = 0.90 / 1.00**

**Hésitations / ambiguïtés :** N5 pose la question de la définition de « prolongée ». Le noyau technique (code en exécution) n'a pas de capacité d'auto-réparation au-delà du reboot ; un état corrompu persiste. Mais le système sociotechnique (code + communauté) est remarquablement résilient aux perturbations prolongées. Le score 0.5 reflète la moyenne entre ces deux lectures, ce qui est cohérent avec le périmètre défini.

**Distinction normativité endogène / imposée :** Mixte, avec dominante endogène. Les normes techniques (priorités d'interruption, politique de scheduling, seuils de page reclaim) sont endogènes — elles émergent de l'architecture du code et de ses contraintes. Les normes de gouvernance (règle « no regressions », hiérarchie des mainteneurs, cycle de release) sont partiellement imposées par des individus (Torvalds, principalement) mais ont été internalisées par la communauté au point de fonctionner même sans intervention directe du BDFL. La norme POSIX est exogène mais sert de contrainte de design, pas de mécanisme de régulation interne.

## A5 — Capacité de révision (formulation V2.1.1 : plasticité endogène)

| Sous-critère                                       | Score | Justification                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| -------------------------------------------------- | ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R1 : Ajustement paramétrique local                 | 1     | Le noyau expose des milliers de paramètres ajustables à chaud : /proc/sys/, sysfs, paramètres de modules, sysctl. Le scheduler est tunable (sched_latency_ns, etc.), le memory manager est paramétrable (vm.swappiness, etc.). Ces ajustements sont locaux et ne modifient pas la structure du système. [1, ch. 17 ; 2, « sysctl »]                                                                                                                                                                                                            |
| R2 : Modification durable de configuration interne | 1     | Kconfig permet de produire des noyaux radicalement différents à partir du même code source (> 15 000 options de configuration). Le chargement/déchargement de modules modifie durablement les capacités du noyau en exécution. Les device trees reconfigurent la perception du matériel. Ces modifications persistent au-delà d'un redémarrage. [1, ch. 17 ; 2, « Kconfig »]                                                                                                                                                                   |
| R3 : Reconfiguration de réseau ou de structure     | 1     | Le noyau peut être profondément restructuré entre versions : remplacement complet du scheduler (O(1) → CFS, 2.6.23), refonte du block layer (introduction de blk-mq), réécriture de sous-systèmes entiers (nouveau pilote DRM). Le processus de merge window permet l'intégration de restructurations majeures à chaque cycle de release (toutes les 8-10 semaines). [3 ; 4]                                                                                                                                                                   |
| R4 : Modification des mécanismes de régulation     | 1     | Les mécanismes de régulation eux-mêmes ont été remplacés plusieurs fois : changement de l'algorithme de scheduling (BFS, CFS, EEVDF), refonte du mécanisme d'OOM kill, remplacement des mécanismes de locking (Big Kernel Lock → fine-grained locking → RCU). La gouvernance a aussi évolué : introduction des mainteneurs de sous-systèmes, adoption de git, création des arbres -next. [1, ch. 4 ; 3 ; 4]                                                                                                                                    |
| R5 : Capacité à produire de nouvelles règles       | 1     | Le noyau a introduit des catégories entièrement nouvelles de fonctionnalités et de régulation : namespaces et cgroups (containerisation — concept absent du design original), eBPF (permet aux utilisateurs d'injecter du code de monitoring/filtrage dans le noyau sans le modifier), LSM framework (permet de brancher des modules de sécurité arbitraires). Ces innovations ne sont pas de simples ajustements : elles créent de nouvelles classes de règles qui n'existaient pas dans le design initial. [2, « eBPF », « namespaces » ; 4] |

**Score A5 = 1.00 / 1.00**

**Hésitations / ambiguïtés :** La question centrale est celle de l'agentivité : est-ce « le noyau » qui se révise ou « les développeurs » qui le révisent ? Dans le périmètre défini (code + architecture + processus de maintenance), le processus de révision est intégré au système lui-même — le mécanisme de merge window, l'arbre des mainteneurs, les mailing lists sont les organes de plasticité du système. R5 pourrait être discuté : eBPF et les namespaces sont des innovations remarquables, mais elles résultent de décisions humaines délibérées, pas d'un processus endogène automatique. Le score 1 se justifie car le périmètre inclut explicitement le processus de maintenance comme composante du système.

### Classification (rempli APRÈS scoring)

- **Régime primaire :** Système adaptatif mature — scores uniformément élevés (≥ 0.90) sur les cinq axes, sans axe déficient. Le profil est celui d'un système complexe pleinement développé avec une capacité d'auto-modification remarquable.
- **Régime secondaire :** Légère dominante plasticité (A1 = A5 = 1.00 vs A2 = A3 = A4 = 0.90) — le système excelle dans sa profondeur structurelle et sa capacité de révision, tandis que les mécanismes de propagation, intégration et normativité sont très légèrement en retrait en raison de limitations structurelles (monolithisme, absence de boucle de rétroaction unique, vulnérabilité aux perturbations prolongées).
- **Marge :** Faible — les scores sont proches du maximum sur tous les axes. La marge d'incertitude se situe principalement sur P4 (isolement) et N5 (résilience prolongée), où un argumentaire différent pourrait faire basculer le score de 0.5 à 0 ou 1, ce qui déplacerait A2 et A4 de ±0.10.
- **Surprise par rapport au jugement intuitif :** Aucune surprise majeure. Le noyau Linux est intuitivement un système très complexe, très intégré et très adaptable, et les scores confirment cette intuition. La relative homogénéité des scores (pas de faiblesse marquée) est notable mais attendue pour un système qui a survécu et évolué pendant 30+ ans avec des milliers de contributeurs.

## Notes libres

**Sur le périmètre « code + architecture + processus de maintenance » :** Ce choix de périmètre est déterminant. Si l'on restreignait le système au seul code en exécution (un noyau sur une machine donnée à un instant t), les scores A5 chuteraient significativement (R3-R5 deviendraient 0 ou 0.5) et A1 perdrait un niveau (la gouvernance). Le choix d'inclure le processus de maintenance comme partie intégrante du système est justifié par le fait que ce processus est formalisé, documenté et inséparable du code qu'il produit — mais il est important de le noter.

**Sur la nature monolithique :** Le choix architectural monolithique (vs. microkernel) a un impact direct sur P4 (isolement) et I4 (rétroaction globale). Un noyau microkernel (type L4, Minix) aurait probablement un P4 plus élevé mais un P5 (couplage) et un I5 (cohérence globale) potentiellement plus bas. Le design monolithique est un choix délibéré qui favorise la performance et la cohérence au détriment de l'isolation.

**Sur eBPF comme marqueur de R5 :** eBPF est probablement l'exemple le plus frappant de R5 dans un système technologique : il permet au noyau de s'auto-instrumenter avec du code arbitraire (mais vérifié) sans modification de sa propre base de code. C'est une forme de métaprogrammation opérationnelle qui crée une catégorie entièrement nouvelle de régulation.

**Comparaison implicite :** Ce système se distinguerait probablement d'un OS propriétaire (Windows kernel) par un A5 plus élevé (plasticité radicale permise par l'open-source) et d'un microkernel par un A2 plus élevé (propagation plus forte due au monolithisme) mais un A3 potentiellement plus bas (coordination plus difficile dans un espace d'adressage partagé).
